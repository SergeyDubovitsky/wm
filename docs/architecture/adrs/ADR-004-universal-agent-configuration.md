# ADR-004: Универсальная конфигурация edge agent и идентификация инстанса

Дата: 2026-03-28  
Статус: accepted

## Контекст

Edge agent должен устанавливаться на слабые локальные компьютеры рядом с объектами автоматизации. Таких объектов и агентов может быть много. При этом:

- на одном хосте потенциально может быть запущено несколько агентов
- агент должен собирать данные из разных типов источников: `KNX`, `Modbus`, `OPC UA`, `DB` и других
- источников одного типа тоже может быть несколько, например `knx_main` и `knx_backup`
- инженер должен заполнять конфиг без ручного управления техническими идентификаторами точек
- конфиги будут копироваться между проектами, поэтому нельзя полагаться только на имя сервиса или hostname
- формат должен одинаково хорошо поддерживать `HTTP` и `MQTT`
- на первом этапе точек немного, но позже конфигурацию нужно будет удобно загружать и в БД

Ранее проект использовал KNX-first схему с файлами `collector.example.yaml` и `group_addresses.example.yaml`. Для мульти-протокольного агента этого недостаточно.

## Решение

### 1. Operational path

Для первых этапов стандартным способом запуска принимается `docker compose`.

- операторский сценарий должен укладываться в две базовые команды: `docker compose up -d` и `docker compose logs -f`
- `Kubernetes` не рассматривается как штатный deployment path для слабых edge-хостов
- альтернативы вроде `systemd`-обертки вокруг контейнера допускаются позже, но не считаются primary operational contract

### 2. Разделение конфигурации

Конфигурация разбивается на три логических слоя:

- `config/agent.example.yaml` — глобальные параметры агента, доставки, storage и observability
- `config/sources.d/*.yaml` — один или несколько source records с параметрами подключения
- `config/points.d/*.yaml` — точки мониторинга, привязанные к `source_id`

Такое разделение позволяет:

- добавлять несколько источников одного типа без разрастания одного большого YAML
- разносить большие наборы точек по нескольким файлам
- проще импортировать source/point records в БД
- держать общие правила на уровне source и не дублировать их в каждой точке

### 3. Идентичность агента

Идентичность разделяется на business и technical уровни:

- `object_id` — обязательный бизнес-идентификатор объекта автоматизации в системе мониторинга
- `agent_id` — технический идентификатор экземпляра агента, который генерируется при первом старте и хранится в `agent.id_file`
- `agent.name` — человекочитаемое имя для диагностики и UI; не считается уникальным ключом

Правила:

- `object_id` должен быть уникален в контуре мониторинга и проверяться во время пусконаладки
- `agent_id` не вычисляется из hostname, MAC-адреса или имени compose service
- при копировании конфигов на другой объект новый агент получает новый `agent_id`, если переносится только конфиг, а не state directory

### 4. Модель источников

Каждый источник обязан иметь:

- уникальный `source_id` в рамках одного агента
- `type`, например `knx`, `modbus`, `opcua`, `db`
- protocol-specific connection settings

Источник может дополнительно задавать `acquisition_defaults` и `publish_defaults`, которые наследуются точками этого источника.

`source_id` используется в конфигурации, валидации и исходящих событиях как идентификатор конкретного подключения, а не только типа протокола.

### 5. Модель точек

Каждая точка мониторинга обязана иметь:

- `point_ref` — технический идентификатор внутри источника
- `name` — человекочитаемое имя
- `value_type` и `value_model`
- `signal_type`

Поля `acquisition` и `publish` на уровне точки рассматриваются как overrides. Базовые значения задаются на уровне source через `acquisition_defaults` и `publish_defaults`.

Правила уникальности:

- `point_ref` должен быть уникален в рамках одного `source_id`
- `name` тоже должен быть уникален в рамках одного `source_id`
- дубликаты считаются ошибкой запуска, а не warning

Примеры `point_ref`:

- для `KNX` — group address
- для `OPC UA` — node id
- для `Modbus` — согласованный register/function reference
- для `DB` — технический идентификатор строки маппинга или query field

### 6. Модель доставки

На первых этапах агент поддерживает один активный sink.

- транспорт выбирается через `delivery.transport`
- допустимые первые варианты: `http` и `mqtt`
- transport-specific параметры хранятся в глобальном конфиге агента

Это сохраняет единый конфигурационный контракт и не заставляет инженера дублировать transport settings в source/point-файлах.

### 7. Унификация правил публикации

Флаг `on_change` исключается из конфигурационной модели.

Остается одно поле `change_threshold`:

- `null` или отсутствие поля означает публикацию при любом изменении значения
- числовое значение означает deadband для аналоговых сигналов
- для `boolean`, `string` и enum-подобных значений числовой `change_threshold` запрещен

### 8. Fail-fast валидация

Агент обязан завершать старт с ошибкой, если обнаружено одно из условий:

- отсутствует `object_id`
- дублируется `source_id`
- дублируется `point_ref` в пределах `source_id`
- дублируется `name` в пределах `source_id`
- point file ссылается на несуществующий `source_id`
- `periodic_interval_seconds` задан некорректно
- `change_threshold` задан для нечислового `value_type`

## Обоснование

- auto-generated persistent `agent_id` защищает от конфликтов при копировании конфигов
- обязательный `object_id` сохраняет бизнес-привязку данных к объекту автоматизации
- `source_id + point_ref` дает универсальную ключевую модель для разных протоколов
- раздельные `agent / sources / points` файлы удобнее для инженера и для будущей загрузки в БД
- `docker compose` дает достаточно простой operational path без Kubernetes и без ручной сборки большого числа команд
- единое поле `change_threshold` делает поведение конфигурации проще и понятнее

## Последствия

Положительные:

- конфиги становятся мульти-протокольными и не завязаны только на KNX
- одна и та же схема работает и для `HTTP`, и для `MQTT`
- появляется надежный способ различать копии конфигов на разных объектах
- облегчается поэтапный переход от YAML к хранению source/point records в БД

Отрицательные:

- для корректной идентичности нужен persistent state directory с `agent_id`
- появляется отдельная дисциплина пусконаладки по контролю уникальности `object_id`
- конфигурация становится многосоставной и требует явного merge/validation этапа на старте

## Отклоненные альтернативы

- Хеш хоста как `agent_id`: не покрывает несколько агентов на одном компьютере и нестабилен при миграции/клонировании.
- Имя сервиса `docker compose` как `agent_id`: слишком сильно связывает identity с launcher и ломается при переименовании сервиса.
- Один большой YAML для всего: удобен только на старте, но быстро становится неудобным для роста и табличного импорта.
- Отдельные `on_change` и `change_threshold`: дублируют друг друга и усложняют валидацию.
